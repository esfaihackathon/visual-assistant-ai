# Build Prompt for Android MVP – “Saral”
*A Voice-First Banking Accessibility Layer for Visually Impaired Users in India*
---
## Product Vision
Build an Android MVP application named **“Saral”** that acts as a **voice-assisted accessibility layer for banking applications**, specifically designed for:
* Fully visually impaired users
* Partially visually impaired users
* Elderly users with accessibility needs
* Users uncomfortable with traditional banking UI navigation
The product should demonstrate how existing Indian banking applications can integrate a lightweight accessibility-first module to provide inclusive banking experiences.
The MVP should focus on:
* Voice-first interactions
* Minimal touch interactions
* Accessibility-compliant navigation
* RBI-aligned secure design principles
* Mock banking workflows for hackathon demonstration
The application must feel realistic, polished, production-inspired, and emotionally reassuring.
---
# Core Product Concept
Saral is NOT a standalone bank.
It is an:
* Accessibility SDK / feature layer
* Voice-guided banking assistant
* Banking accessibility framework
that can be integrated into existing Indian banking apps like:
* State Bank of India
* HDFC Bank
* ICICI Bank
* Axis Bank
The MVP will simulate banking operations using mocked APIs and local stateless data.
---
# Primary Goal of MVP
Demonstrate an accessibility-first banking experience where a visually impaired user can:
1. Open app
2. Authenticate using fingerprint biometric
3. Hear voice greeting
4. Navigate completely using voice
5. Check balance
6. Transfer money
7. Request cheque book
8. Hear confirmations clearly
9. Operate with minimal visual dependency

# Key MVP Principles
## 1. Voice First
Every major action should:
* Speak aloud
* Accept voice input
* Give audio confirmations
## 2. Accessibility First
Use:
* Android TalkBack compatibility
* Large typography
* High contrast mode
* Minimal screen clutter
* Large tappable areas
* Haptic feedback
## 3. Security Aware
Even though backend is mocked:
* Never read full account numbers aloud
* Never speak sensitive authentication data
* Require confirmation before transfers
* Simulate RBI-compliant UX practices
## 4. Stateless Architecture
No real banking integration.
Use:
* Mock JSON data
* Local in-memory session
* No persistent sensitive storage

# RBI & Banking UX Alignment (Important)
Even though this is a hackathon MVP, the UX should demonstrate awareness of Indian banking regulations and secure practices.
## Simulated RBI-Compliant Practices
### Do NOT:
* Speak full account number
* Speak sensitive identifiers
* Auto-confirm transfers
### Must:
* Require biometric login
* Require confirmation before transfer
* Provide audible transaction confirmation
* Auto logout after inactivity
* Use secure, reassuring language
